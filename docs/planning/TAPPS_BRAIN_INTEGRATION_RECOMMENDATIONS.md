## Executive Summary

tapps-brain is a production-grade, deterministic memory engine with 33 modules, 28 MCP tools, 6 configurable profiles, multi-agent Hive, cross-project federation, bi-temporal versioning, cognitive science-inspired decay, and 1,226 tests. TappsMCP currently exposes approximately 40% of these capabilities through a thin re-export shim layer in tapps-core.

The 2026 industry landscape reveals three urgent priorities:

1. **Memory security is now an OWASP Top 10 item** (ASI06). Microsoft documented 50 real-world poisoning attacks in 60 days. tapps-brain has safety primitives but they are not enforced by default.
2. **The multi-agent memory problem is formalized**. ICLR 2026 MemAgents workshop and multiple arxiv papers define the taxonomy. tapps-brain's Hive is well-positioned but not wired into TappsMCP's Agent Teams.
3. **Hybrid retrieval (vectors + graphs + BM25) is consensus**. tapps-brain has all three engines but they are loosely coupled and not enabled by default.

This document details 18 recommendations across 4 priority tiers, with effort estimates, implementation guidance, and the 2026 research that motivates each one.

## 2026 Industry Landscape

### Memory Taxonomy (ICLR 2026 MemAgents Workshop)

The research community has converged on a three-axis taxonomy for agent memory:

| Axis | Categories | tapps-brain Mapping |
|---|---|---|
| **Functional** | Factual (knowledge), Experiential (insights/skills), Working (active context) | architectural/pattern = factual, procedural = experiential, context = working |
| **Dynamics** | Formation (extraction), Evolution (consolidation/forgetting), Retrieval (access) | `extract_durable_facts` = formation, consolidation + decay = evolution, BM25/recall = retrieval |
| **Scope** | Private, Shared, Federated | project/session = private, hive = shared, federation = federated |

tapps-brain's 4-tier model covers all three axes. The `procedural` tier (added in Epic 65.11) correctly anticipated the formal distinction between experiential and factual memory.

### Three Architectural Camps

| Camp | Representatives | Architecture | Strengths | Weaknesses |
|---|---|---|---|---|
| **LLM-managed** | Letta | Agent decides what to remember/forget via tiered context | Flexible, adaptive | Expensive, non-deterministic, vendor lock-in |
| **Graph-first** | Zep (Graphiti), Mem0 | Temporal knowledge graph + entity resolution | Rich relationships, temporal reasoning | Complex extraction pipelines, often cloud-dependent |
| **Deterministic/local** | tapps-brain, Vestige, brain-mcp | Rule-based extraction, local SQLite, no LLM calls | Predictable, fast, zero-cost, reproducible | Less semantic depth for complex relationships |

**2026 consensus:** Hybrid is best. Vectors for breadth, graphs for depth, rules for speed. tapps-brain already has all three engines (BM25 + FAISS vectors + relation triples) but they are loosely coupled.

### Memory Security (OWASP ASI06)

OWASP added **ASI06: Memory & Context Poisoning** to the Top 10 for Agentic Applications 2026. Key findings:

- Microsoft documented **50 real-world poisoning attempts** across 31 companies in 60 days
- Companies embed hidden instructions in "Summarize with AI" buttons to inject persistent memory
- **MINJA attack achieves >95% injection success** on unprotected stores
- Attack and execution are **temporally decoupled** -- injection in February, damage in April
- **OWASP Agent Memory Guard** is the reference implementation: SHA-256 integrity baselines, injection detection, sensitive data leakage, rapid-change anomaly detection

### Multi-Agent Memory

Two foundational 2026 papers formalize multi-agent memory:

- **Collaborative Memory (arXiv 2505.18279):** Bipartite access control graphs linking users, agents, and resources. Two memory tiers: private fragments (originator-only) and shared fragments (selective access).
- **Multi-Agent Memory from Computer Architecture (arXiv 2603.10062):** Three-layer hierarchy (I/O, cache, memory). Two critical protocol gaps: cache sharing across agents and structured memory access control.

tapps-brain's Hive (namespace isolation, agent registry, 4 conflict resolution policies) aligns with both papers.

### MCP Ecosystem Trends

- **1,864+ MCP servers** in the ecosystem as of March 2026
- **Modular composition > monolithic** -- enterprises maintain curated catalogs like microservices
- **Streamable HTTP** unlocked remote deployments but surfaced stateful session challenges
- The official MCP memory server uses a simple knowledge graph (entities + relations + observations)

### Cognitive Science Integration

Vestige (Rust, MCP server) has raised the bar by implementing FSRS-6 spaced repetition (21 parameters), prediction error gating, synaptic tagging, and memory dreaming. tapps-brain already implements desirable difficulty (Roediger & Karpicke 2006) and stability growth (Jost's First Law) but does not market them as prominently.

### Enterprise Validation

AWS Bedrock AgentCore Memory launched with two-tier architecture (short-term + long-term), three extraction strategies (semantic, user preference, summary), hierarchical namespaces, and streaming notifications for memory changes. This validates tapps-brain's tiered approach at enterprise scale.

## Architecture Decision: Gateway Model

### The Problem

Three access paths exist today with no clear guidance:

| Path | Entry Point | Features |
|---|---|---|
| **A. tapps-brain direct** | `tapps-brain-mcp` (28 tools) or Python `MemoryStore` | Full feature set |
| **B. Through TappsMCP** | `tapps_memory` (28 actions) | Pipeline-integrated but incomplete |
| **C. Both** | Two MCP servers | Everything, but split-brain risk |

### The Decision

**TappsMCP is the gateway for code projects. tapps-brain is the standalone engine for everything else.**

```
Code project using TappsMCP  -->  tapps_memory (single MCP tool)  -->  tapps-brain
Non-code project             -->  tapps-brain-mcp (28 tools)       -->  tapps-brain
Python library consumer      -->  tapps-brain (import directly)    -->  tapps-brain
```

**Rules:**
1. Never run both `tapps-brain-mcp` and TappsMCP on the same project (split-brain risk)
2. TappsMCP must close the feature gap so projects never need tapps-brain-mcp alongside it
3. Hive and Federation use separate databases (`~/.tapps-brain/hive/`, `~/.tapps-brain/memory/federated.db`) and are safe to share across gateways
4. TappsMCP's value-add over raw tapps-brain is pipeline integration: auto-recall, auto-capture, quality-informed memory, profile auto-detection

### Documentation Update

AGENTS.md and ONBOARDING.md should include:

> **Memory:** TappsMCP includes a built-in persistent memory system powered by tapps-brain. Use `tapps_memory` for all memory operations. Do not run `tapps-brain-mcp` as a separate server alongside TappsMCP. If you need standalone memory without code quality tools, use `tapps-brain-mcp` directly instead.

## Performance Constraints

Several recommendations add overhead to hot paths (save, recall, search). These targets prevent compounding latency from multiple features:

| Hot Path | Added By | Latency Budget |
|----------|----------|----------------|
| `save()` | R01 (safety check), R02 (HMAC hash), R03 (rate limiter) | < 5ms combined additional overhead |
| `recall()`/`search()` | R02 (hash verify), R04 (trust scoring), R10 (graph boost) | < 10ms combined additional overhead |

**Measurement protocol:**
- Benchmark before/after each phase with the standard 1000-entry test store
- If any phase exceeds budget, defer lower-priority additions until optimization is complete
- Track p50 and p99 latencies, not just averages

## Recommendations

### P0: Security & OWASP Compliance

These are non-negotiable given the 2026 threat landscape. OWASP ASI06 compliance is becoming an enterprise purchasing criterion.

---

#### R01: Enforce Safety Checks on Every Memory Write

**Motivation:** tapps-brain has 6 prompt injection detection patterns in `safety.py` but they are opt-in. The MINJA attack achieves >95% success on unprotected stores. Memory poisoning is temporally decoupled -- the injection and its damage can be weeks apart.

**What to do:**
- Make `check_content()` a mandatory step in `MemoryStore.save()` (tapps-brain change)
- Blocked writes return `SafetyCheckResult` with flagged patterns and a clear error
- Add a `safety_bypass=True` escape hatch for trusted internal writes (e.g., system seeding)
- Surface safety results in `tapps_memory` response when a write is blocked
- Add new action: `tapps_memory(action="safety_check", value="...")` for pre-flight validation
- **Enforcement rollout:** Ship with a `safety.enforcement: warn | block` profile setting. Default to `warn` for the first release (log flagged writes but allow them), switch default to `block` after validating false positive rates across pilot users.

> **Bypass access control:** `safety_bypass=True` must NOT be honorable by any caller. It is restricted to:
> - Writes where `source` is `"system"` (not `"agent"` or `"inferred"`)
> - OR when the project owner has set `memory.safety.allow_bypass: true` in `.tapps-mcp.yaml`
>
> Agent-sourced writes must never self-bypass, regardless of parameters passed. The bypass exists for system seeding and migration scripts, not for runtime agent use. If an agent needs to save content that triggers safety checks, the content should be reviewed — not the safety check bypassed.

**Where to change:**
- `tapps-brain`: `store.py` (save method, bypass access control), `safety.py` (ensure patterns are current)
- `tapps-mcp`: `server_memory_tools.py` (surface safety results, enforce bypass restrictions)
- Profile YAML: new `safety.enforcement` setting (`warn` | `block`)

**Effort:** Low (1-2 days). The safety module already exists; this wires it into the write path with access control.

**Success Criteria:**
- 0 false negatives against the MINJA test suite (6 known injection patterns)
- < 0.1% false positive rate on the existing tapps-brain test corpus
- Agent-sourced writes cannot bypass safety checks regardless of parameters
- No measurable latency regression on save operations (< 1ms additional)

**2026 reference:** OWASP Agent Memory Guard, Microsoft AI Recommendation Poisoning report, Palo Alto Unit42 persistent memory poisoning research.

---

#### R02: Add Cryptographic Integrity Baselines

**Motivation:** OWASP Agent Memory Guard specifies keyed hashing per memory entry to detect tampering. If an external process or attack modifies the SQLite database directly, integrity checks catch it.

> **Design note:** Plain SHA-256 is insufficient for adversarial tampering detection. An attacker with direct SQLite access can recompute SHA-256 hashes after modifying data. HMAC-SHA256 with a secret key stored outside the database ensures that hash recomputation requires possession of the key, not just database access.

**What to do:**
- Add `integrity_hash` column to `memories` table (HMAC-SHA256 of `key + value + tier + source_agent + created_at`)
- Use HMAC-SHA256 with a 256-bit random secret key stored outside the database at `~/.tapps-brain/integrity.key`
- Auto-generate the key on first use; if the key file is missing, generate a new one and re-baseline all entries on next `verify_integrity()` call
- Compute HMAC on save, verify on read
- Add `verify_integrity()` method to MemoryStore that scans all entries
- Surface integrity violations in `memory_health()` and `tapps_session_start`
- New action: `tapps_memory(action="verify_integrity")` for on-demand checking
- **Migration:** Add `integrity_hash` as a nullable column. Populate hashes lazily on next write/reinforce, not via a bulk migration. Entries without hashes skip verification until written.

**Where to change:**
- `tapps-brain`: `persistence.py` (schema migration), `store.py` (HMAC on save/verify on read), new `integrity.py` (key management)
- `tapps-mcp`: `server_memory_tools.py` (new action), `server_pipeline_tools.py` (session start enrichment)

**Effort:** Medium (2-3 days). Requires schema migration, key management, and backward compatibility.

**Success Criteria:**
- 100% tamper detection rate on modified entries (adversarial test: modify DB directly, verify catches it)
- < 2ms additional latency on read path (HMAC verify)
- Lazy population completes within 1 session for stores with < 10,000 entries
- Key loss recovery via `verify_integrity(rebaseline=True)` completes in < 30s for 10,000 entries

**2026 reference:** OWASP Agent Memory Guard integrity baseline specification.

#### R03: Add Write Rate Limiting and Anomaly Detection

**Motivation:** Rapid-fire writes are a signal of automated poisoning. Normal agent behavior is 1-5 writes per session. 50+ writes in a minute is anomalous.

**What to do:**
- Add a sliding window rate limiter to `MemoryStore.save()` (configurable per profile)
- Default: max 20 writes per minute, max 100 writes per session
- Exceeding the limit triggers a warning in the response (not a hard block, to avoid breaking legitimate batch operations)
- Log anomalies to the audit trail with `anomaly_type: "rapid_write"`
- Surface anomaly counts in `memory_health()`

**Batch-mode exemption:** Operations that legitimately produce many writes in a short window (`import_markdown`, `seed`, `federation_sync`, `consolidate`) acquire an internal `batch_context` that:
- Suspends rate limit warnings for the duration of the batch
- Logs a single audit entry: `"batch_write": {"operation": "import_markdown", "count": 127, "duration_ms": 340}`
- Resets the sliding window after the batch completes
- `batch_context` is acquired internally by known batch operations, **not** exposed as a user-facing parameter (to prevent abuse as a rate-limit bypass)

**Where to change:**
- `tapps-brain`: `store.py` (rate limiter, batch context), `audit.py` (anomaly + batch logging)
- Profile YAML: new `safety.rate_limit` section

**Effort:** Low-Medium (1-2 days).

**Success Criteria:**
- 0 false positives during normal agent sessions (1-5 writes/session)
- Anomaly logging fires on > 20 writes/minute synthetic test
- No impact on batch operations using `batch_context` exemption
- Rate limit state resets cleanly across sessions

**2026 reference:** OWASP Agent Memory Guard rapid-change anomaly detection.

#### R04: Add Trust-Scored Retrieval

**Motivation:** Not all memory sources are equally trustworthy. A human-saved architectural decision should outweigh an agent-inferred context note during retrieval. Current retrieval weights confidence (time-decayed) but not source trustworthiness.

> **Priority note:** Trust-scored retrieval is a defense-in-depth ranking improvement — it makes poisoned entries rank lower but does not prevent them from being saved or detect that they exist. It is complementary to R01-R03 (which actively block/detect attacks) and ships alongside the P1 feature enablement phase.

**What to do:**
- Add `source_trust` as a 5th signal in composite scoring (alongside relevance, confidence, recency, frequency)
- Default trust scores: `human: 1.0`, `system: 0.9`, `agent: 0.7`, `inferred: 0.5`
- Make trust scores configurable per profile in YAML
- Adjust composite scoring formula: `score = w_relevance * R + w_confidence * C + w_recency * T + w_frequency * F + w_trust * S`

**Where to change:**
- `tapps-brain`: `retrieval.py` (add trust signal), `models.py` (trust weight in profile), profile YAMLs
- No TappsMCP changes needed (flows through automatically)

**Effort:** Medium (2-3 days). Touches the scoring core.

**Success Criteria:**
- Human-sourced entries rank higher than agent-sourced entries of equal relevance in A/B test
- No ranking regression for single-source stores (all entries same trust level)
- < 1ms additional latency on composite scoring

**2026 reference:** OWASP defense framework: "trust-aware retrieval -- weight results by source trustworthiness."

#### R05: Expose Profile Management via MCP

**Motivation:** tapps-brain ships 6 built-in profiles optimized for different use cases (repo-brain, personal-assistant, customer-support, research-knowledge, project-management, home-automation). TappsMCP hardcodes `repo-brain`. Users cannot view, switch, or customize profiles through MCP.

**What to do:**
- Add three new actions to `tapps_memory`:
  - `tapps_memory(action="profile_info")` -- show active profile layers, decay config, scoring weights
  - `tapps_memory(action="profile_switch", value="research-knowledge")` -- switch active profile
  - `tapps_memory(action="profile_list")` -- list available profiles with one-line descriptions
- Profile switch persists to `.tapps-brain/profile.yaml` in the project root

**Where to change:**
- `tapps-mcp`: `server_memory_tools.py` (3 new action handlers)
- `tapps-core`: `memory/injection.py` (respect active profile on injection config)

**Effort:** Low (1 day). tapps-brain already implements all profile operations.

#### R06: Profile Auto-Detection at Session Start

**Motivation:** Different projects need different memory decay curves. A research project's "established facts" shouldn't decay at the same rate as a web app's "deployment procedures." TappsMCP already runs `tapps_project_profile()` to detect tech stack -- use that signal.

**What to do:**
- In `tapps_session_start`, after detecting project type, auto-select the best profile:
  - Python web API / general code repo --> `repo-brain` (default)
  - Data science / ML / research project --> `research-knowledge`
  - Monorepo with CI/PM artifacts --> `project-management`
  - Custom `.tapps-brain/profile.yaml` present --> always wins
- Include active profile name and key characteristics in session start response
- Allow override via `.tapps-mcp.yaml` setting: `memory.profile: "research-knowledge"`

**Where to change:**
- `tapps-mcp`: `server_pipeline_tools.py` (session start logic)
- `tapps-mcp`: `config/settings.py` (add `memory.profile` setting)

**Effort:** Low (1 day).

#### R07: Enable Promotion Engine by Default

**Motivation:** Currently all memories stay in whatever tier they are saved to forever. tapps-brain has a full promotion/demotion engine that auto-promotes frequently accessed memories to longer-lived tiers and demotes stale ones. This makes memory self-organizing. The 2026 consensus is that static memory tiers are insufficient -- memories should flow based on usage.

**What to do:**
- Enable promotion/demotion in the default `repo-brain` profile (it exists but is not active by default)
- Default promotion thresholds:
  - context --> procedural: 3 accesses, 7 days age, confidence >= 0.3
  - procedural --> pattern: 5 accesses, 14 days age, confidence >= 0.4
  - pattern --> architectural: 10 accesses, 30 days age, confidence >= 0.5
- Demotion: confidence at floor for > half-life period with no access
- Run promotion/demotion check on `reinforce()` calls
- Surface promotions/demotions in `tapps_memory` responses: `"promoted: context -> procedural"`

**Where to change:**
- `tapps-brain`: `profiles/repo-brain.yaml` (enable promotion rules)
- `tapps-brain`: `store.py` or `promotion.py` (ensure check runs on reinforce)
- `tapps-mcp`: `server_memory_tools.py` (surface promotion events in response)

**Effort:** Low (1 day). The engine exists; this enables and surfaces it.

#### R08: Expose Hive for Agent Teams

**Motivation:** When Claude Code Agent Teams are enabled (`CLAUDE_CODE_EXPERIMENTAL_AGENT_TEAMS=1`), multiple agents work on the same project with no shared memory. Each agent rediscovers context independently. tapps-brain's Hive provides namespace-isolated multi-agent memory with 4 conflict resolution policies.

**What to do:**
- Detect Agent Teams env var in `tapps_session_start`
- Auto-initialize HiveStore and register the current agent
- Add new actions to `tapps_memory`:
  - `tapps_memory(action="hive_search", value="query")` -- search across all Hive namespaces
  - `tapps_memory(action="hive_propagate")` -- push local architectural-tier memories to Hive
  - `tapps_memory(action="hive_status")` -- show Hive namespaces, agent count, entry counts
  - `tapps_memory(action="agent_register", key="agent-id", value="agent-name")`
- Configure auto-propagation per profile:
  - `auto_propagate_tiers: ["architectural"]` (share system decisions)
  - `private_tiers: ["context"]` (never share session ephemera)

**Privacy safeguards:**
- Before auto-propagating to Hive, check that the entry does not have `scope: "private"` or user-defined sensitive tags (configurable: `hive.sensitive_tags: ["credentials", "personal", "secret"]`)
- Surface a first-run notice on first Hive initialization: "Agent Teams memory sharing is enabled. Architectural-tier memories will be visible to all agents in this project."
- Add `tapps_memory(action="hive_config")` to let users view and modify propagation rules at runtime

**Where to change:**
- `tapps-mcp`: `server_pipeline_tools.py` (session start Hive init, first-run notice)
- `tapps-mcp`: `server_memory_tools.py` (5 new action handlers, including `hive_config`)
- `tapps-mcp`: `server_helpers.py` (HiveStore singleton management, sensitive-tag filtering)

**Effort:** Medium (3-4 days). Requires HiveStore lifecycle management, privacy filtering, and testing.

**2026 reference:** Collaborative Memory (arXiv 2505.18279) bipartite access control, Multi-Agent Memory from Computer Architecture (arXiv 2603.10062) cache sharing protocol.

#### R09: Markdown Import for Onboarding

**Motivation:** Many projects already have a `MEMORY.md` file (including this monorepo). That knowledge exists as flat text that Claude reads but cannot search, rank, or decay. tapps-brain has `markdown_import.py` that parses markdown into structured memory entries.

- Deduplicate against existing entries (skip if key already exists with higher confidence)
- Optionally integrate into `tapps_init` flow: "Found MEMORY.md -- import into memory store?"

**Where to change:**
- `tapps-mcp`: `server_memory_tools.py` (new action handler)
- `tapps-mcp`: `pipeline/init.py` (optional init integration)

**Effort:** Low (1 day). tapps-brain already implements the parser.

#### R10: Enable Graph-Boosted Recall by Default

**Motivation:** The 2026 consensus is "hybrid RAG -- vectors for breadth, graphs for depth." tapps-brain has graph-boosted recall (`use_graph_boost=True`, `graph_boost_factor=0.15`) that boosts scores of entries connected via relation triples. It is implemented but disabled by default.

**What to do:**
- Enable `use_graph_boost=True` only when the store contains **>= 10 relation triples**. Below that threshold, skip the graph query entirely to avoid unnecessary overhead on new projects.
- Set default `graph_boost_factor=0.1` (conservative, can be tuned per profile)
- Make the threshold configurable per profile: `retrieval.graph_boost_min_relations: 10`
- Ensure relation extraction runs on save (it currently does for some patterns)
- Add relation count and activation status to `memory_health()` output:
  - `"Graph-boosted recall: inactive (3/10 relations)"` or
  - `"Graph-boosted recall: active (47 relations, boost=0.1)"`
- When the threshold is crossed for the first time, log an info message: "Graph-boosted recall activated — 10+ relation triples available"

**Where to change:**
- `tapps-brain`: `retrieval.py` (density-gated activation, default parameter change)
- `tapps-brain`: profile YAMLs (add `retrieval.graph_boost: true`, `retrieval.graph_boost_min_relations: 10`)

**Effort:** Low (half day). The code exists; this adds a threshold gate and flips defaults.

**2026 reference:** Mem0 graph memory architecture, Vector vs Graph RAG consensus (MachineLearningMastery, Vectorize).

#### R12: Bi-Temporal Versioning for Architecture Decisions

**Motivation:** When an architectural-tier memory is updated, the old value is overwritten. tapps-brain supports `supersede()` which preserves the full version chain with `valid_at`/`invalid_at` timestamps. This is critical for understanding when and why decisions changed.

**What to do:**
- When `tapps_memory(action="save", tier="architectural")` targets an existing key, automatically use `supersede()` instead of overwrite
- Preserve full version chain in the store
- Promote the `history` action in pipeline guidance: "Use `tapps_memory(action='history', key='...')` to see decision evolution"
- Include version count in search results for architectural entries

**Where to change:**
- `tapps-mcp`: `server_memory_tools.py` (auto-supersede logic for architectural tier)

**Effort:** Low (1 day). tapps-brain already implements supersede/history.

#### R13: Memory-Informed Scoring Context

**Motivation:** Scoring today is stateless -- it applies generic rules without project-specific knowledge. If the memory store has an architectural decision like "we use SQLAlchemy 2.0 async sessions exclusively," the scorer could flag sync SQLAlchemy usage as a convention violation.

**What to do:**
- Before scoring, recall architectural-tier memories tagged with relevant keywords (e.g., "conventions", "standards", "patterns")
- Pass recalled conventions as additional context to the scoring engine
- Add a new scoring category: "Project Conventions" (weight configurable, default low)
- Flag violations as suggestions, not hard failures (since conventions can change)

**Where to change:**
- `tapps-mcp`: `scoring/scorer.py` (accept optional convention context)
- `tapps-mcp`: `server_helpers.py` (recall conventions before scoring)
- `tapps-mcp`: `scoring/categories.py` (new category definition)

**Effort:** High (4-5 days). Touches the scoring core and requires careful design to avoid false positives.

**Note:** This is the highest-impact P2 recommendation but also the most complex. Consider starting with a minimal version that only flags exact string matches against known conventions.

#### R14: Auto-Save Quality Results to Memory

**Motivation:** When `tapps_consult_expert` returns a domain recommendation or `tapps_quick_check` finds a recurring pattern, that knowledge should persist across sessions. Currently, quality results are ephemeral -- they exist only in the conversation context.

**What to do:**
- After `tapps_consult_expert`, if the response contains actionable recommendations, auto-save to memory as `pattern` tier with tag `expert-{domain}`
- After `tapps_quick_check`, if the same file fails the same category 3+ times (tracked via metrics), save a procedural memory: "File X has recurring {category} issues"
- Use `source: "auto-quality"` (distinct from manual `agent` saves) and `scope: "project"` for auto-saved entries
- Configurable via `.tapps-mcp.yaml`: `memory.auto_save_quality: true` (default: true)

**Privacy safeguards:**
- Auto-saved entries are tagged `source: "auto-quality"` so users can distinguish them from explicit saves
- Surface auto-saved entry count and most recent entries in `memory_health()` output
- Support filtering: `tapps_memory(action="list", source="auto-quality")` for auditing what was auto-saved
- **Opt-out** via `memory.auto_save_quality: false` in `.tapps-mcp.yaml` — this setting should be prominently documented in onboarding guidance

**Where to change:**
- `tapps-mcp`: `server_pipeline_tools.py` (expert consultation handler)
- `tapps-mcp`: `server_helpers.py` (quick_check result processing, health reporting)
- `tapps-mcp`: `server_memory_tools.py` (source filter support in list action)
- `tapps-mcp`: `config/settings.py` (new setting)

**Effort:** Medium (2-3 days).

#### R16: Prediction Error Gating

**Motivation:** Vestige implements prediction error gating -- novel information that contradicts existing knowledge gets higher storage priority. tapps-brain already has contradiction detection. Extending it to prioritize contradictory saves would implement this cognitive science principle.

- Surface contradiction detection in `tapps_memory` save responses: "Note: this contradicts existing entry '{key}'"

**Where to change:**

### tapps-brain Unique Differentiators

1. **Only deterministic memory system with configurable profiles** -- no other framework offers 6 built-in profiles with customizable decay, promotion, and scoring
2. **Only system with both Hive (multi-agent) and Federation (cross-project)** -- competitors offer one or neither

1. **Entity resolution** -- Mem0 and Zep extract and resolve entities ("Alice" = "our CTO"). tapps-brain has basic regex relation extraction (7 patterns)

4. **Dashboard/visualization** -- Vestige has a 3D neural visualization dashboard. tapps-brain has CLI + metrics but no visual interface

### Migration & Rollback Strategy

Existing users with tapps-brain databases must upgrade safely. Each phase should be feature-flagged via profile YAML so any individual feature can be disabled without code changes.

**R01 (Safety enforcement):**
- Ship with `safety.enforcement: warn` (default). Log flagged writes but allow them.
- Monitor false positive rate across pilot users before switching default to `block`.
- If false positive rate exceeds 0.1%, tune patterns before enabling enforcement.

**R02 (Schema migration):**
- Add `integrity_hash` as a **nullable** column. No bulk migration required.
- Populate hashes lazily on next write or reinforce. Entries without hashes skip verification until written.
- If the HMAC key file is lost, generate a new one and re-baseline via `verify_integrity(rebaseline=True)`.

**R03 (Rate limiting):**
- Batch operations (`import_markdown`, `seed`, `federation_sync`) acquire an internal `batch_context` that suspends rate limiting for the batch duration and logs a single audit entry.
- `batch_context` is not exposed as a user-facing parameter to prevent abuse.

**General rollback:**
- Each recommendation is gated by a profile setting (e.g., `safety.enabled: true`, `integrity.enabled: true`).
- Disabling a feature via profile YAML takes effect immediately on next operation — no restart required.
- Schema additions (nullable columns) are backward-compatible and do not require rollback.
