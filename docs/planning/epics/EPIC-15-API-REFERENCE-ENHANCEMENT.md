# Epic 15: API Reference Enhancement

**Status:** Open
**Priority:** P1
**Estimated LOE:** ~2–2.5 weeks
**Dependencies:** DocsMCP Epic 4 (API Documentation Generation)
**Blocks:** None

---

## Goal

Improve DocsMCP API documentation generation to include full docstrings (not just summaries), structured return schemas, per-subpackage splitting, noise filtering, and better example extraction.

## Current State

API docs generated by DocsMCP suffer from:
1. **Truncated docstrings** — `api_docs.py` line 350 uses `parsed.summary` only, never `parsed.description`
2. **No return schemas** — `APIDocFunction.returns` is bare `str`, no structured type info
3. **145KB monoliths** — entire package dumped into single file
4. **Noise** — logger constants, re-export modules, private functions included
5. **Weak examples** — `_find_examples()` regex too narrow, `_MAX_EXAMPLES = 2`

## Acceptance Criteria

- [ ] Full docstring body rendered (summary + description + params + returns + raises)
- [ ] Return type annotations parsed and displayed in API docs
- [ ] Large packages split into per-subpackage files with index
- [ ] Logger constants and re-export-only modules filtered out by default
- [ ] Example extraction covers doctests, `>>>` patterns, and test files
- [ ] All existing DocsMCP tests pass; new tests for each story

---

## Stories

### 15.1 — Full Docstring Rendering

**Priority:** Critical | **Points:** 3

**Source Files:**
- `packages/docs-mcp/src/docs_mcp/generators/api_docs.py` — rendering logic
- `packages/docs-mcp/src/docs_mcp/extractors/docstring_parser.py` — parsed output

**Tasks:**
- Use `parsed.description` (full body) in addition to `parsed.summary`
- Render Args, Returns, Raises, Examples sections from parsed docstring fields
- Preserve code blocks and markdown formatting from original docstrings
- Add tests with multi-line docstrings, Google-style, and NumPy-style

**Definition of Done:** Generated API docs show full docstring content, not just first line.

---

### 15.2 — Structured Return Schemas

**Priority:** High | **Points:** 3

**Source Files:**
- `packages/docs-mcp/src/docs_mcp/generators/api_docs.py` — return type rendering
- `packages/docs-mcp/src/docs_mcp/extractors/python.py` — AST extraction

**Tasks:**
- Parse return type annotations from AST (e.g., `-> dict[str, Any]`, `-> list[str]`)
- Add `return_type` field to API doc function model
- Render return type in generated docs
- For simple dict returns, note the type annotation; complex schema extraction is future work

**Definition of Done:** API docs show return type annotations from function signatures.

---

### 15.3 — Per-Subpackage API Splitting

**Priority:** High | **Points:** 5

**Source Files:**
- `packages/docs-mcp/src/docs_mcp/generators/api_docs.py` — splitting logic
- `packages/docs-mcp/src/docs_mcp/server_gen_tools.py` — tool handler

**Tasks:**
- When generating for a package, group modules by subpackage (first directory under package root)
- Generate separate markdown file per subpackage
- Generate index file linking to all subpackage docs
- Add `scope` parameter: `"module"` (single file), `"package"` (subpackage), `"project"` (all)
- Respect existing single-file behavior when `scope="project"` or package has no subpackages

**Definition of Done:** TappsMCP API docs split into ~8 focused files instead of one 145KB monolith.

---

### 15.4 — Noise Filtering

**Priority:** Medium | **Points:** 3

**Source Files:**
- `packages/docs-mcp/src/docs_mcp/generators/api_docs.py` — filtering logic
- `packages/docs-mcp/src/docs_mcp/extractors/python.py` — extraction filtering

**Tasks:**
- Filter `logger = structlog.get_logger()` and similar constant assignments
- Detect re-export-only `__init__.py` files (only `from X import Y` statements) and skip or mark
- Filter private functions (`_prefix`) unless they have public docstrings
- Add `include_private` parameter (default `False`)
- Add `include_reexports` parameter (default `False`)

**Definition of Done:** Generated API docs exclude noise by default; options available to include.

---

### 15.5 — Example Extraction Enhancement

**Priority:** Medium | **Points:** 4

**Source Files:**
- `packages/docs-mcp/src/docs_mcp/generators/api_docs.py` — `_find_examples()`

**Tasks:**
- Expand regex to match: doctest (`>>> `), `# Example:` blocks, inline code blocks in docstrings
- Increase `_MAX_EXAMPLES` from 2 to 5
- Extract usage examples from test files matching `test_<function_name>` pattern
- Render examples with `python` syntax highlighting markers

**Definition of Done:** API docs include richer examples from docstrings and test files.

## Risk Assessment

| Risk | Impact | Mitigation |
|------|--------|------------|
| Docstring parser edge cases | Medium | Test with Google, NumPy, and reST styles |
| Per-subpackage splitting breaks existing consumers | Medium | Default behavior unchanged; splitting is opt-in via scope |
| Test file example extraction too noisy | Low | Limit to first N lines of test function body |
