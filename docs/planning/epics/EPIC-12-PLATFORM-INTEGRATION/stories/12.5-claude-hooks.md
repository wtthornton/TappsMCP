# Story 12.5: Claude Code Hooks Generation

**Status:** Pending
**Tier:** 2
**Priority:** P1
**LOE:** 3-4 days
**Platform:** Claude Code

---

## Summary

Claude Code supports 17 lifecycle hook events that fire at specific points in the
session: startup, tool use, agent completion, compaction, and more. These hooks
allow scripts to inject context, enforce quality gates, and prevent premature
session termination. Currently `tapps_init` bootstraps TappsMCP in consuming
projects but does not configure any Claude Code hooks, leaving quality enforcement
entirely dependent on the LLM following instructions voluntarily. This story adds
hook generation to the `tapps_init` flow so that every project initialized with
TappsMCP gets a set of `.claude/settings.json` hook entries and companion shell
scripts in `.claude/hooks/` that enforce the TappsMCP workflow automatically —
injecting context at session start, triggering quality reminders after file edits,
backing up scoring context before compaction, injecting TappsMCP awareness into
subagents, and hard-blocking session stop and task completion until
`tapps_validate_changed` has been called.

## Acceptance Criteria

- `tapps_init` generates `.claude/settings.json` with a `hooks` key containing
  entries for all seven TappsMCP hooks described below
- If `.claude/settings.json` already exists with a `hooks` key, the generated
  hooks are merged into the existing array rather than overwriting it
- `tapps_init` generates the following hook shell scripts in `.claude/hooks/`:
  - `tapps-session-start.sh`
  - `tapps-session-compact.sh`
  - `tapps-post-edit.sh`
  - `tapps-stop.sh`
  - `tapps-task-completed.sh`
  - `tapps-pre-compact.sh`
  - `tapps-subagent-start.sh`
- All generated scripts are created with executable permissions (`chmod +x`)
- `tapps-stop.sh` reads `stop_hook_active` from the stdin JSON and exits 0
  (no-op) when it is `true`, preventing infinite loops
- `tapps-stop.sh` exits 2 with an explanatory message to stderr when validation
  has not yet been run, blocking the session from stopping
- `tapps-task-completed.sh` exits 2 with an explanatory message when quality
  gates have not passed, blocking task completion
- The `SessionStart` hook with matchers `startup` and `resume` calls
  `tapps_session_start` and injects TappsMCP context into Claude's context
- The `SessionStart` hook with matcher `compact` re-injects TappsMCP context
  after context compaction
- The `PostToolUse` hook with matchers `Edit` and `Write` outputs a reminder
  to run `tapps_quick_check` on the modified file
- The `PreCompact` hook backs up the current scoring context to a temporary file
- The `SubagentStart` hook injects a brief TappsMCP awareness message into every
  spawned subagent's context
- A unit test verifies the hooks config structure in the generated
  `.claude/settings.json`
- A unit test verifies each hook script file is created and is executable
- A unit test verifies `tapps-stop.sh` contains a check for `stop_hook_active`
- `mypy --strict` passes with no new errors
- `ruff check` passes with no new errors

### Settings.json structure

The generated `hooks` block in `.claude/settings.json` follows this structure:

```json
{
  "hooks": {
    "SessionStart": [
      {
        "matcher": "startup|resume",
        "hooks": [{ "type": "command", "command": ".claude/hooks/tapps-session-start.sh" }]
      },
      {
        "matcher": "compact",
        "hooks": [{ "type": "command", "command": ".claude/hooks/tapps-session-compact.sh" }]
      }
    ],
    "PostToolUse": [
      {
        "matcher": "Edit|Write",
        "hooks": [{ "type": "command", "command": ".claude/hooks/tapps-post-edit.sh" }]
      }
    ],
    "Stop": [
      {
        "hooks": [{ "type": "command", "command": ".claude/hooks/tapps-stop.sh" }]
      }
    ],

      {
        "hooks": [{ "type": "command", "command": ".claude/hooks/tapps-task-completed.sh" }]
```

### Stop hook infinite-loop prevention

The `tapps-stop.sh` script must parse `stop_hook_active` from the stdin JSON
before any other logic. When `stop_hook_active` is `true`, the script must exit 0
immediately and produce no output. Failure to do this causes an infinite loop
where every exit-2 triggers another Stop event.

```bash
#!/usr/bin/env bash
INPUT=$(cat)
ACTIVE=$(echo "$INPUT" | python3 -c "import sys,json; d=json.load(sys.stdin); print(d.get('stop_hook_active','false'))" 2>/dev/null)
if [ "$ACTIVE" = "True" ] || [ "$ACTIVE" = "true" ]; then
  exit 0
fi
# Enforcement logic: check validation state, exit 2 if not validated
echo "Run tapps_validate_changed before ending the session." >&2
exit 2
```

### Merge strategy for existing settings.json

1. If `.claude/settings.json` does not exist, write a fresh file containing only
   the generated `hooks` block (plus `$schema` and any other fields `tapps_init`
   already writes).
2. If it does exist, load the existing JSON, deep-merge the generated `hooks`
   entries into `settings["hooks"]` for each event key, and write the result
   back. Do not overwrite entries that are already present for a given matcher.

### Approach

1. In `pipeline/init.py`, add a `_generate_claude_hooks(project_root: Path) -> None`
   function that:
   a. Reads hook script templates from `src/tapps_mcp/prompts/hooks/`
   b. Creates `.claude/hooks/` directory if it does not exist
   c. Writes each script and calls `os.chmod(path, 0o755)`
   d. Reads or initializes `.claude/settings.json`
   e. Merges the hooks config block
   f. Writes the updated settings file
2. Call `_generate_claude_hooks(project_root)` from the main `tapps_init` handler,
   guarded by a check that the platform is Claude Code (or always run it, since
   `.claude/` is Claude Code-specific).
3. Store hook script templates as plain `.sh` files under
   `src/tapps_mcp/prompts/hooks/` so they are included in the package via
   `pyproject.toml` `[tool.setuptools.package-data]`.
