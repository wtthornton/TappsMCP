# Story 12.4: Environment Variables in MCP Config

**Status:** Pending
**Tier:** 1
**Priority:** P0
**LOE:** 0.5 day
**Platform:** All (Claude Code, Cursor, VS Code)

---

## Summary

TappsMCP uses `TAPPS_MCP_PROJECT_ROOT` to determine which directory to sandbox all
file operations to. Currently the generated MCP configs produced by
`distribution/setup_generator.py` omit an `env` block entirely, leaving the server
to auto-detect the project root at startup. Auto-detection is fragile: it can
silently resolve to the wrong directory when the MCP client launches the server
process from a different working directory than the user's project. Adding an
explicit `"env": {"TAPPS_MCP_PROJECT_ROOT": "${workspaceFolder}"}` to every
generated config guarantees that the path is always set correctly, regardless of
how the client spawns the process. All three supported platforms use the same env
key name and the same `${workspaceFolder}` interpolation token.

## Acceptance Criteria

- The Claude Code config (`.mcp.json`) generated by `setup_generator.py` includes
  `"env": {"TAPPS_MCP_PROJECT_ROOT": "${workspaceFolder}"}` inside the `tapps-mcp`
  server entry
- The Cursor config (`.cursor/mcp.json`) generated by `setup_generator.py` includes
  `"env": {"TAPPS_MCP_PROJECT_ROOT": "${workspaceFolder}"}` inside the `tapps-mcp`
  server entry
- The VS Code config (`.vscode/mcp.json`) generated by `setup_generator.py`
  includes `"env": {"TAPPS_MCP_PROJECT_ROOT": "${workspaceFolder}"}` inside the
  `tapps-mcp` server entry (alongside the existing `"type": "stdio"` field)
- The `"command"` and `"args"` fields in the server entry are not modified or
  removed
- When `setup_generator.py` merges into an existing config file that already
  contains an `env` block for `tapps-mcp`, the merge replaces the `tapps-mcp` entry
  (existing merge behavior) — no partial-env merge edge cases are introduced
- A unit test verifies the `env` field is present and contains
  `TAPPS_MCP_PROJECT_ROOT` in the output for all three platforms
- `mypy --strict` passes with no new errors
- `ruff check` passes with no new errors

## Implementation Notes

### Files to change

| File | Change |
|------|---------|
| `src/tapps_mcp/distribution/setup_generator.py` | Add `"env"` block to `_TAPPS_SERVER_ENTRY` or create per-platform entries |

### Approach

1. Open `setup_generator.py` and locate the `_TAPPS_SERVER_ENTRY` dict and the
   `_HOST_CONFIGS` dict (around line 24–45).

2. The simplest correct change is to add `"env"` directly to `_TAPPS_SERVER_ENTRY`,
   since all three platforms use the same key name and value:

   ```python
   _TAPPS_SERVER_ENTRY: dict[str, Any] = {
       "command": "tapps-mcp",
       "args": ["serve"],
       "env": {
           "TAPPS_MCP_PROJECT_ROOT": "${workspaceFolder}",
       },
   }
   ```

   Because `_HOST_CONFIGS` and `_merge_config` both reference `_TAPPS_SERVER_ENTRY`
   (or a copy of it), this single change propagates to all three platform configs
   with no further edits.

3. If any platform requires a different entry structure (e.g., the VS Code config
   already adds a `"type": "stdio"` field in a separate step), verify that the
   `"type"` field is not lost after adding `"env"`. Check whether `"type"` is
   injected during generation or whether it is already part of the VS Code-specific
   template in `_HOST_CONFIGS`. If it is only in `_HOST_CONFIGS` and not in
   `_TAPPS_SERVER_ENTRY`, no conflict exists.

4. Do not introduce per-platform `_TAPPS_SERVER_ENTRY` variants unless the audit
   in step 3 reveals a genuine conflict. Keeping a single shared entry is simpler
   and reduces the risk of the platforms drifting out of sync.

5. Run `uv run tapps-mcp init --host claude-code --dry-run` (if a dry-run flag
   exists) or generate configs to a temp directory and inspect the JSON to confirm
   the `env` block appears correctly before writing tests.

## Testing Requirements

### Unit tests — `tests/unit/test_setup_generator.py` (existing file, add cases)

- **Claude Code env field**: Call the Claude Code config generation path. Assert
  `output["mcpServers"]["tapps-mcp"]["env"]["TAPPS_MCP_PROJECT_ROOT"]` equals
  `"${workspaceFolder}"`.

- **Cursor env field**: Call the Cursor config generation path. Assert
  `output["mcpServers"]["tapps-mcp"]["env"]["TAPPS_MCP_PROJECT_ROOT"]` equals
  `"${workspaceFolder}"`.

- **VS Code env field**: Call the VS Code config generation path. Assert
  `output["servers"]["tapps-mcp"]["env"]["TAPPS_MCP_PROJECT_ROOT"]` equals
  `"${workspaceFolder}"`.

- **Existing fields preserved**: For each platform, assert that `"command"` and
  `"args"` are still present in the server entry alongside `"env"`.

- **VS Code type field preserved**: Assert that `output["servers"]["tapps-mcp"]`
  still contains `"type": "stdio"` after the env field is added (regression guard).

- **Merge preserves other servers**: Provide an existing config dict with a
  different server entry. Assert the merge result still contains both the existing
  server and the updated `tapps-mcp` entry with the `env` block.
