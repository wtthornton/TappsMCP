# Story 12.11: Cursor Rule Types Enhancement

**Status:** Complete
**Tier:** 3
**Priority:** P2
**LOE:** 1 day
**Platform:** Cursor

---

## Summary

TappsMCP currently generates only `alwaysApply: true` rules for Cursor, which
means all TappsMCP guidance is injected into every conversation regardless of
relevance. Cursor supports three additional rule types that are far more
targeted: `autoAttach` rules activate automatically when matching files are
referenced in the conversation, `agentRequested` rules are loaded on-demand
when the agent judges them relevant based on the `description` field, and
`manual` rules require an explicit `@`-mention. This story extends the Cursor
rule generation in `distribution/setup_generator.py` to produce all three
appropriate rule types: an `autoAttach` rule with `globs: "*.py"` that injects
scoring standards whenever Python files are in scope, an `agentRequested` rule
with a description that enables on-demand expert consultation, and retains the
existing `alwaysApply` rule for core pipeline instructions that should always
be present. Generating the right rule type for each concern reduces context
bloat and makes each rule fire at the most appropriate moment.

## Acceptance Criteria

- `tapps_init` and `tapps-mcp init` generate three distinct `.mdc` rule files
  in `.cursor/rules/` rather than one
- `tapps-pipeline.mdc` uses `alwaysApply: true` with no `globs` or `description`,
  covering core pipeline instructions (session start, validate before completing)
- `tapps-python-quality.mdc` uses `globs: "*.py"` and `alwaysApply: false` with
  no `description`, activating automatically when Python files are referenced
- `tapps-expert-consultation.mdc` uses only a `description` field (no `globs`,
  `alwaysApply: false`), making it an agentRequested rule that the agent loads
  when expert guidance is relevant
- Each `.mdc` file has valid YAML frontmatter (parseable as YAML between `---`
  delimiters)
- `tapps-pipeline.mdc` frontmatter contains exactly: `alwaysApply: true`
- `tapps-python-quality.mdc` frontmatter contains `globs: "*.py"` and
  `alwaysApply: false`
- `tapps-expert-consultation.mdc` frontmatter contains a non-empty `description`
  string and no `globs` or `alwaysApply` keys
- The body of `tapps-python-quality.mdc` instructs the agent to call
  `tapps_quick_check` after editing Python files and references the 7 scoring
  categories
- The body of `tapps-expert-consultation.mdc` explains how to invoke
  `tapps_consult_expert` and lists available expert domains
- All three rules are generated by both `tapps_init` (MCP tool) and
  `tapps-mcp init` (CLI command)
- Unit tests verify the frontmatter of each generated rule file
- `mypy --strict` passes with no new errors
- `ruff check` passes with no new errors

## Implementation Notes

### Files to change

| File | Change |
|------|--------|
| `src/tapps_mcp/distribution/setup_generator.py` | Replace single rule generation with three-rule generation |
| `src/tapps_mcp/pipeline/init.py` | Update `tapps_init` to write all three rule files |
| `src/tapps_mcp/prompts/cursor/` | Add rule body templates for each rule type |

### Rule frontmatter by type

**Always rule (`tapps-pipeline.mdc`):**

```yaml
---
alwaysApply: true
---
```

Body: Core pipeline workflow â€” start session with `tapps_session_start`,
run `tapps_quick_check` after edits, run `tapps_validate_changed` before
declaring work complete.

**Auto Attached rule (`tapps-python-quality.mdc`):**

```yaml
---
globs: "*.py"
alwaysApply: false
---
```

Body: Python-specific scoring standards. Reference all 7 scoring categories
(correctness, security, maintainability, performance, documentation, testing,
style). Instruct agent to call `tapps_quick_check` when Python files are
referenced and scores below 70 need attention.

**Agent Requested rule (`tapps-expert-consultation.mdc`):**

```yaml
---
description: "TappsMCP domain expert consultation â€” use when needing guidance on security, performance, architecture, testing, or other domain-specific best practices."
---
```

Body: Instructions for calling `tapps_consult_expert` with the appropriate
domain. List available expert domains: security, performance, architecture,
testing, documentation, accessibility, devops, database, api, frontend,
backend, data-science, ml, cloud, mobile, embedded.

### Approach

1. Define three rule template strings as constants or Jinja2 templates in
   `src/tapps_mcp/prompts/cursor/rules/`.
2. Refactor `setup_generator.py` to call a `generate_cursor_rules(output_dir)`
   function that writes all three files.
3. Ensure `pipeline/init.py` calls the same function when `tapps_init` runs.
4. Do not remove the existing `tapps-pipeline.mdc` generation path â€” replace
   it. Projects that already have the old file should have it overwritten on
   the next `tapps_init` run (idempotent).

### Rule precedence note

The three rules operate at different token costs. `alwaysApply` rules cost
tokens on every prompt. `autoAttach` rules cost tokens only when matching files
are open. `agentRequested` rules cost tokens only when the agent deems them
relevant. The three-rule split minimizes total token usage compared to a single
`alwaysApply` rule containing all content.
