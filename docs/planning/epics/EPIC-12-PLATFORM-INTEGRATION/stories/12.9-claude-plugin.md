# Story 12.9: Claude Code Plugin Bundle

**Status:** Pending
**Tier:** 3
**Priority:** P2
**LOE:** 2 days
**Platform:** Claude Code

---

## Summary

Claude Code supports a plugin format that bundles MCP configuration, subagent
definitions, skills, and hooks into a single installable directory structure
rooted at `.claude-plugin/`. Publishing TappsMCP as a Claude Code plugin allows
users to install all quality tooling in a single step via `cursor://install-plugin`
or a deep link, rather than running `tapps_init` and manually wiring each
component. This story creates the `.claude-plugin/` directory structure inside
the TappsMCP repository so it can be distributed alongside the Python package,
bundles the outputs from stories 12.5 (hooks), 12.6 (subagent definitions), and
12.8 (skills) into the plugin, and adds a `plugin.json` manifest with the
required name, version, and description fields.

  `name`, `version`, and `description`

- `plugin.json` `version` field matches the version in `pyproject.toml`
- The plugin directory contains an `agents/` subdirectory with the three
  TappsMCP subagent definition files:
  - `agents/tapps-reviewer.md`
  - `agents/tapps-researcher.md`
  - `agents/tapps-validator.md`
- The plugin directory contains a `skills/` subdirectory with TappsMCP skill
  definitions from story 12.8 (at minimum `tapps-score/SKILL.md`,
  `tapps-gate/SKILL.md`, and `tapps-validate/SKILL.md`)

  configuration including the `instructions` field from story 12.2
- The plugin directory contains a `README.md` describing installation and usage
- A unit test verifies the plugin directory structure is complete (all required
  files and directories present)

  fields with non-empty values

### plugin.json format

```json
{
  "name": "tapps-mcp",
  "version": "1.0.0",
  "description": "Code quality scoring, security scanning, and quality gates"
}
```

The `name` field must be lowercase kebab-case. The `version` field must be kept
in sync with the version in `pyproject.toml`; consider reading it dynamically
from a shared constant or generating the file as part of the build process.

### Approach

1. Create the `plugin/claude/` directory tree in the repository.
2. Store the `plugin.json` manifest as a static file; add a CI check that
   verifies the version matches `pyproject.toml`.
3. Share agent and skill templates with the `pipeline/init.py` generator by
   referencing the same source files from `src/tapps_mcp/prompts/`.
4. The `hooks/hooks.json` in the plugin directory uses the same hook script
   references as `.claude/settings.json`, but scoped to the plugin's `hooks/`
   directory.
5. Include the entire `plugin/` directory in `MANIFEST.in` and `pyproject.toml`
   `[tool.setuptools.package-data]` so it is distributed with the package.

### hooks.json within plugin

The plugin-scoped `hooks.json` follows the same schema as `.claude/settings.json`
hooks but uses relative paths within the plugin directory:

```json
{
  "hooks": {
    "SessionStart": [
      {
        "matcher": "startup|resume",
        "hooks": [{ "type": "command", "command": "hooks/tapps-session-start.sh" }]
      }
    ],
    "Stop": [
      {
        "hooks": [{ "type": "command", "command": "hooks/tapps-stop.sh" }]
      }
    ]
  }
}
```
