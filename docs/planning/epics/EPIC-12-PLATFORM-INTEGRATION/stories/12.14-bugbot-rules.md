# Story 12.14: Cursor BugBot Rules

**Status:** Pending
**Tier:** 4
**Priority:** P3
**LOE:** 0.5 day
**Platform:** Cursor

---

## Summary

Cursor BugBot is an automated PR review system that processes over 2 million
pull requests per month and achieves a 70% resolution rate on the issues it
finds. BugBot reads `.cursor/BUGBOT.md` for project-specific quality standards
and uses directory hierarchy inheritance — it walks up from the changed files
to find the nearest applicable `BUGBOT.md`. Currently TappsMCP's `tapps_init`
generates no BugBot configuration. This story adds generation of
`.cursor/BUGBOT.md` to the `tapps_init` flow, embedding TappsMCP's quality
standards, security requirements, and Python style rules directly into BugBot's
review criteria. Since BugBot cannot call MCP tools, the `BUGBOT.md` must
encode the standards as explicit rules rather than tool invocations.

## Acceptance Criteria

- `tapps_init` generates `.cursor/BUGBOT.md` in the target project directory
- `tapps-mcp init` generates `.cursor/BUGBOT.md` when Cursor is the selected
  platform
- The generated file is valid markdown
- The file includes a "Code Quality Standards" section covering:
  - Minimum acceptable score thresholds (e.g., overall score >= 70)
  - The 7 scoring categories and what constitutes a failure in each
- The file includes a "Security Requirements" section covering:
  - No hardcoded secrets, passwords, or API keys
  - No use of `eval()`, `exec()`, or `pickle.loads()` on untrusted input
  - SQL queries must use parameterized statements
  - File paths must be validated before use
- The file includes a "Python Style Rules" section covering:
  - Type annotations required on all public functions and methods
  - Docstrings required on all public classes and functions
  - No use of bare `except:` clauses
  - Maximum function complexity (cyclomatic complexity <= 10)
- The file includes a "Testing Requirements" section covering:
  - New functions must have corresponding test coverage
  - Tests must not use production credentials or external services without mocking
- The file references TappsMCP scoring categories so BugBot reviewers have
  context on what the categories mean
- The generation is idempotent: re-running `tapps_init` overwrites the file
- A unit test verifies the file is created with expected section headings
- A unit test verifies the file contains security requirement text
- `mypy --strict` passes with no new errors
- `ruff check` passes with no new errors

### Generated file content

```markdown
# TappsMCP Quality Standards for BugBot

This project uses TappsMCP (Code Quality MCP Server) for automated quality
analysis. The following standards are enforced during PR review.

## Code Quality Standards

All Python files must meet TappsMCP scoring thresholds:
- Overall score: >= 70 (development), >= 80 (staging), >= 90 (production)
- No individual category score below 50

### Scoring Categories

| Category | What BugBot Should Check |
|----------|-------------------------|
| Correctness | Logic errors, unchecked return values, unreachable code |
| Security | Hardcoded secrets, unsafe deserialization, injection vulnerabilities |
| Maintainability | Functions > 50 lines, cyclomatic complexity > 10, god classes |
| Performance | Nested loops on large data, synchronous I/O in async context |
| Documentation | Missing docstrings on public API, outdated parameter descriptions |
| Testing | Functions without test coverage, tests that call real external services |
| Style | Inconsistent naming, bare `except`, missing type annotations |

## Security Requirements

Flag any of the following as blocking issues:
- Hardcoded passwords, API keys, tokens, or secrets (any string matching key/secret/password/token patterns)
- Use of `eval()` or `exec()` with non-literal arguments
- `pickle.loads()` on data from external sources
- Raw SQL string concatenation (use parameterized queries)
- File path operations without validation against an allowed base directory
- `subprocess` calls with `shell=True` and interpolated user input

## Python Style Rules

Flag the following as non-blocking warnings:
- Public functions and methods without type annotations on parameters and return value
- Public classes and functions without docstrings
- Bare `except:` clauses (must specify exception type)
- Functions with cyclomatic complexity > 10
- Functions longer than 50 lines (excluding docstrings and blank lines)
- Mutable default arguments in function signatures

## Testing Requirements

Flag the following as non-blocking warnings:
- New public functions without a corresponding test in `tests/`
- Tests that make real HTTP requests without `responses` or `httpx` mocking
- Tests that read from or write to production configuration files
- Tests that depend on specific environment variables without explicit fixtures

## Directory Hierarchy

This `BUGBOT.md` applies to all files in `.cursor/` and all subdirectories.
Place a subdirectory `BUGBOT.md` to override these rules for specific
sub-packages with different thresholds.
```

### Approach

1. Add a `prompts/cursor/BUGBOT.md` template file.
2. Add a `generate_bugbot_rules(output_dir: Path) -> Path` function to
   `setup_generator.py` that writes the template to
   `output_dir / ".cursor" / "BUGBOT.md"`.
3. Call `generate_bugbot_rules` from `pipeline/init.py` when the target
   platform includes Cursor.
4. Since `tapps_init` is platform-agnostic by default, generate `BUGBOT.md`
   whenever `.cursor/` is being generated (i.e., alongside `mcp.json` and
   hooks for Cursor).

### BugBot constraints to document

BugBot cannot invoke MCP tools. The `BUGBOT.md` must encode all quality
standards as explicit, self-contained prose rules. Do not reference
`tapps_quick_check` as something BugBot will call — instead, translate the
tool's criteria into declarative rules BugBot can apply directly.
