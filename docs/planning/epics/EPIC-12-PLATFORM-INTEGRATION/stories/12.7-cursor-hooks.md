# Story 12.7: Cursor Hooks Generation

**Status:** Complete
**Tier:** 2
**Priority:** P1
**LOE:** 2 days
**Platform:** Cursor

---

## Summary

Cursor supports a lightweight hook system defined in a single `.cursor/hooks.json`
file. Unlike Claude Code's 17-event model, Cursor exposes six events (`beforeMCPExecution`,
`afterMCPExecution`, `beforeFileEdit`, `afterFileEdit`, `start`, `stop`) with a
simpler command-only execution model and no blocking semantics except via
`followup_message` on the `stop` event. Currently `tapps_init` does not generate
any Cursor hook configuration, leaving Cursor users without the automated quality
enforcement that Claude Code users receive. This story adds Cursor hook generation
to the `tapps_init` flow so that Cursor-initialized projects get a `.cursor/hooks.json`
file and companion shell scripts that log TappsMCP tool calls, trigger quality check
reminders after file edits, and surface a `tapps_validate_changed` prompt when the
agent stops.

---

## Research Reference

See `../research/cursor-features.md` for the complete Cursor hook specification,

shapes for each event, the distinction between fire-and-forget events and
`followup_message`-capable events, and the known limitation that `afterFileEdit`
cannot block session flow.

## Acceptance Criteria

- `tapps_init` with `--platform cursor` (or when Cursor is detected) generates
  `.cursor/hooks.json` with entries for the three TappsMCP hooks described below
- If `.cursor/hooks.json` already exists, the generated hook entries are merged
  into the existing array rather than overwriting it
- `tapps_init` generates the following hook shell scripts in `.cursor/hooks/`:
  - `tapps-before-mcp.sh`
  - `tapps-after-edit.sh`
  - `tapps-stop.sh`
- All generated scripts are created with executable permissions (`chmod +x`)
- The `beforeMCPExecution` hook logs the MCP tool name being called and injects
  a brief TappsMCP context reminder via stdout
- The `afterFileEdit` hook outputs a reminder to run `tapps_quick_check` on the
  modified file (fire-and-forget; does not block)
- The `stop` hook uses `followup_message` in its output JSON to prompt the agent
  to run `tapps_validate_changed` before the session ends
- The `stop` hook script does not attempt to block via exit code (Cursor does not
  support exit-2 blocking on the `stop` event)
- A unit test verifies `.cursor/hooks.json` is created with the correct schema
- A unit test verifies each hook script file is created and is executable
- A unit test verifies `tapps-stop.sh` outputs a `followup_message` JSON field
  rather than exiting with code 2
- A unit test verifies `tapps-after-edit.sh` does not contain `exit 2`
- `mypy --strict` passes with no new errors
- `ruff check` passes with no new errors

### hooks.json structure

The generated `.cursor/hooks.json` follows this structure:

```json
{
  "hooks": [
    {
      "event": "beforeMCPExecution",
      "command": ".cursor/hooks/tapps-before-mcp.sh"
    },
    {
      "event": "afterFileEdit",
      "command": ".cursor/hooks/tapps-after-edit.sh"
    },
    {
      "event": "stop",
      "command": ".cursor/hooks/tapps-stop.sh"
    }
  ]
}
```

### Event semantics and constraints

Cursor hooks differ from Claude Code hooks in two important ways:

1. **No blocking via exit codes.** Cursor ignores non-zero exit codes from hook
   scripts. The only way to surface enforcement to the agent after the `stop` event
   is to emit a JSON object with a `followup_message` field on stdout. The agent
   receives this as an additional message and may act on it, but it does not
   hard-block the session the way Claude Code's exit-2 does.

2. **`afterFileEdit` is fire-and-forget.** The hook runs after the edit is complete
   and cannot influence what happens next. Use stdout text output only; do not
   attempt to write a `followup_message` field (it is ignored for this event).

### Stop hook output format

The `tapps-stop.sh` script must write a JSON object to stdout when it wants to
surface a message to the agent:

```bash
#!/usr/bin/env bash
# Read the stop event payload from stdin (unused but consumed to avoid pipe errors)
INPUT=$(cat)
# Output a followup_message to prompt the agent to validate before ending
echo '{"followup_message": "Before ending: please run tapps_validate_changed to confirm all changed files pass quality gates."}'
exit 0
```

### beforeMCPExecution hook

The `tapps-before-mcp.sh` script receives a JSON payload on stdin that includes
the MCP tool name being called. It should log the call for observability and
optionally inject a context reminder:

```bash
#!/usr/bin/env bash
INPUT=$(cat)
TOOL=$(echo "$INPUT" | python3 -c "import sys,json; d=json.load(sys.stdin); print(d.get('tool','unknown'))" 2>/dev/null)
echo "[TappsMCP] MCP tool invoked: $TOOL" >&2
exit 0
```

### afterFileEdit hook

The `tapps-after-edit.sh` script receives a JSON payload on stdin that includes
the edited file path. It should output a plain-text reminder on stdout:

```bash
#!/usr/bin/env bash
INPUT=$(cat)
FILE=$(echo "$INPUT" | python3 -c "import sys,json; d=json.load(sys.stdin); print(d.get('file','unknown'))" 2>/dev/null)
echo "File edited: $FILE — consider running tapps_quick_check to verify quality."
exit 0
```

### Merge strategy for existing hooks.json

1. If `.cursor/hooks.json` does not exist, write a fresh file with the three
   hook entries.
2. If it does exist, load the existing JSON, check the `hooks` array for
   existing entries with the same `event` value, and append only those entries
   whose `event` is not already present. Do not overwrite existing entries for
   the same event.

### Approach

1. In `pipeline/init.py`, add a
   `_generate_cursor_hooks(project_root: Path) -> None` function that:
   a. Reads hook script templates from `src/tapps_mcp/prompts/cursor_hooks/`
   b. Creates `.cursor/hooks/` directory if it does not exist
   c. Writes each script and calls `os.chmod(path, 0o755)`
   d. Reads or initializes `.cursor/hooks.json`
   e. Merges the hook entries using the event-keyed merge strategy
   f. Writes the updated hooks file
2. Call `_generate_cursor_hooks(project_root)` from the main `tapps_init` handler
   when the platform is `cursor` (determined from the `tapps_init` arguments).
3. Store hook script templates as plain `.sh` files under
   `src/tapps_mcp/prompts/cursor_hooks/` so they are included in the package via
   `pyproject.toml` `[tool.setuptools.package-data]`.
