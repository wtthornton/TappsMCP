# Story 12.2: Server Instructions Field

**Status:** Pending
**Tier:** 1
**Priority:** P0
**LOE:** 0.5 day
**Platform:** Claude Code

---

## Summary

Claude Code uses lazy loading ("Tool Search") to manage tool counts when a user
has many MCP servers registered. When a server is deferred, Claude Code relies on
the `instructions` field in the server's MCP config entry to decide whether the
server's tools are relevant to the current task. TappsMCP currently generates MCP
config entries without an `instructions` field in `distribution/setup_generator.py`,
making TappsMCP tools invisible during deferred loading. Adding a concise
`instructions` string to the generated Claude Code config (`.mcp.json`) ensures
TappsMCP is always discovered by Tool Search and its tools are surfaced when
performing code quality, security, or documentation tasks.

## Acceptance Criteria

- The Claude Code config generated by `setup_generator.py` includes an
  `instructions` field at the server level (same level as `command`, `args`, `env`)
- The `instructions` value accurately describes TappsMCP's capabilities in plain
  English and is suitable for semantic matching by Claude Code's Tool Search
- The `instructions` field is not added to Cursor or VS Code configs (those
  platforms do not use this field)
- Existing generated config fields (`command`, `args`) are not modified or removed
- A unit test verifies the generated Claude Code config contains the `instructions`
  key with a non-empty string value
- `mypy --strict` passes with no new errors
- `ruff check` passes with no new errors

## Implementation Notes

### Files to change

| File | Change |
|------|--------|
| `src/tapps_mcp/distribution/setup_generator.py` | Add `instructions` field to the Claude Code MCP config template |

### Approach

1. Locate the Claude Code config template or dict-building logic in
   `setup_generator.py`. The generated output is a `.mcp.json` file with a
   `mcpServers` key.
2. Add `"instructions"` as a sibling key to `"command"` and `"args"` inside the
   `tapps-mcp` server entry. Use the value from the research reference:

```json
{
  "mcpServers": {
    "tapps-mcp": {
      "command": "uvx",
      "args": ["tapps-mcp", "serve"],
      "env": {
        "TAPPS_MCP_PROJECT_ROOT": "${workspaceFolder}"
      },
      "instructions": "Code quality scoring (0-100 across 7 categories), security scanning (Bandit + secret detection), quality gates (pass/fail against configurable presets), documentation lookup, domain expert consultation, and project profiling for Python projects."
    }
  }
}
```

3. The `instructions` field is Claude Code-specific. Do not add it to the Cursor
   (`.cursor/mcp.json`) or VS Code (`.vscode/mcp.json`) config templates.
4. If the setup generator uses a shared base dict for all platforms and then
   customizes per-platform, add the `instructions` field in the Claude Code-specific
   branch only.

## Testing Requirements

### Unit tests — `tests/unit/test_setup_generator.py` (existing file, add cases)

- Call the Claude Code config generation path and capture the output dict or JSON.
- Assert the top-level structure is `{"mcpServers": {"tapps-mcp": {...}}}`.
- Assert `output["mcpServers"]["tapps-mcp"]["instructions"]` exists and is a
  non-empty string.
- Assert the instructions string contains at least one of the key capability
  phrases (e.g., "quality", "scoring", "security").
- Assert the Cursor config output does NOT contain an `instructions` key inside
  the server entry.
- Assert the VS Code config output does NOT contain an `instructions` key inside
  the server entry.

### Regression

- Run the full test suite (`uv run pytest tests/ -v`) and confirm no regressions.
