# TappsMCP Consumer Requirements

This is the canonical checklist of what a consuming project needs for TappsMCP tools to work. Use it to verify your setup or diagnose issues. For AI agents: check these requirements when TappsMCP tools fail or are unavailable.

---

## 1. TappsMCP server available to the agent

**What:** The tapps-mcp MCP server must appear in the host's list of available MCP servers. Tools like `tapps_session_start` and `tapps_quick_check` only work when the server is connected and running.

**Why:** MCP tools are not global -- they are only callable when the host has loaded and started the server process.

**How to set up:**
- **Claude Code:** Add to `.mcp.json` (project-scoped) or `~/.claude.json` (user-scoped, not recommended). Run `tapps-mcp init` to auto-generate.
- **Cursor:** Add to `.cursor/mcp.json`. Run `tapps-mcp init --host cursor`.
- **VS Code:** Add to `.vscode/mcp.json`. Run `tapps-mcp init --host vscode`.

**How to verify:** See the host-specific checks in the [verification table](#verification-table) below.

---

## 2. MCP config in the project

**What:** The host must load a config file that defines the tapps-mcp server entry (command, args, env). This is the file that tells the host how to start the server.

**Why:** Without this config, the host does not know tapps-mcp exists and will not start it.

**Config file per host:**

| Host | Config file | Generated by |
|------|------------|--------------|
| Claude Code | `.mcp.json` (project) or `~/.claude.json` (user) | `tapps-mcp init` |
| Cursor | `.cursor/mcp.json` | `tapps-mcp init --host cursor` |
| VS Code | `.vscode/mcp.json` | `tapps-mcp init --host vscode` |

**Typical entry (stdio transport):**
```json
{
  "mcpServers": {
    "tapps-mcp": {
      "command": "uvx",
      "args": ["tapps-mcp", "serve"],
      "env": {}
    }
  }
}
```

**How to verify:** `tapps-mcp doctor` checks for MCP config files and reports missing ones.

---

## 3. Tool permissions (Claude Code only)

**What:** Claude Code requires explicit permission rules to auto-approve MCP tool calls. Without them, every tool call prompts for user approval.

**Why:** Claude Code's permission model blocks unknown tools by default. Adding both the bare server name and wildcard ensures all tools are auto-approved across Claude Code versions.

**How to set up:** Ensure `.claude/settings.json` contains:
```json
{
  "permissions": {
    "allow": [
      "mcp__tapps-mcp",
      "mcp__tapps-mcp__*"
    ]
  }
}
```

The bare `mcp__tapps-mcp` entry is needed as a fallback -- the wildcard `mcp__tapps-mcp__*` has known issues in some Claude Code versions. Run `tapps-mcp init` or `tapps-mcp upgrade` to generate this automatically.

**Cursor / VS Code:** These hosts manage MCP permissions differently. No `.claude/settings.json` needed.

**How to verify:** `tapps-mcp doctor` checks `.claude/settings.json` for the correct permission entries.

## 4. Bootstrap (init)

**What:** Run `tapps-mcp init` (CLI) or `tapps_init` (MCP tool) once per project. This creates AGENTS.md, TECH_STACK.md, platform rules, hooks, skills, and subagents. It also seeds project context so tools like `tapps_project_profile` and `tapps_consult_expert` have data to work with.

**Why:** Without init, the agent has no AGENTS.md instructions, no platform hooks to enforce quality, and no project profile for context-aware recommendations.

**How to set up:**
```bash
tapps-mcp init                        # auto-detect host
tapps-mcp init --host claude-code     # target specific host
```
Or via MCP tool: `tapps_init()` (call `tapps_session_start` first).

**How to verify:** Check that `AGENTS.md` and `TECH_STACK.md` exist in the project root. `tapps-mcp doctor` checks for these files.

---

## 5. Python (for scoring tools)

**What:** Scoring, quality gate, and security scan tools (`tapps_score_file`, `tapps_quality_gate`, `tapps_quick_check`, `tapps_security_scan`, `tapps_validate_changed`) target Python files. They require Python source files to analyze.

**Why:** The scoring pipeline uses ruff, mypy, bandit, and radon -- all Python-specific tools. Non-Python projects can still use documentation lookup, expert consultation, memory, config validation, and project context tools.

**Tools that work without Python files:**
- `tapps_lookup_docs`, `tapps_consult_expert`, `tapps_research`
- `tapps_validate_config` (Dockerfile, docker-compose, etc.)
- `tapps_memory`, `tapps_session_notes`
- `tapps_project_profile`, `tapps_impact_analysis`
- `tapps_init`, `tapps_doctor`, `tapps_upgrade`

**How to verify:** `tapps-mcp doctor` checks for installed quality tools (ruff, mypy, bandit, radon).

---

## 6. CLI fallback

**What:** When MCP tools are not available (server not running, host not configured, or permission issues), use the `tapps-mcp` CLI directly for essential operations.

**Why:** The CLI provides the same core functionality as the MCP tools for init, upgrade, and diagnostics. It works regardless of MCP host state.

**Key CLI commands:**
```bash
tapps-mcp init                  # bootstrap project
tapps-mcp upgrade               # refresh generated files after version update
tapps-mcp doctor                # diagnose configuration issues
tapps-mcp doctor --quick        # fast mode (skip tool version checks)
tapps-mcp build-plugin          # generate Claude Code plugin directory
tapps-mcp rollback --list       # list available upgrade backups
```

**How to verify:** Run `tapps-mcp doctor` -- if it runs successfully, the CLI is available.

---

## 7. Verification table

| # | Requirement | How to verify | Notes |
|---|-------------|---------------|-------|
| 1 | Server available | **Claude Code:** `/mcp` lists tapps-mcp. **Cursor:** Settings > MCP shows it. **VS Code:** MCP panel shows it. | If missing, run `tapps-mcp init` and restart host |
| 2 | MCP config exists | `tapps-mcp doctor` reports MCP config status | Check `.mcp.json`, `.cursor/mcp.json`, or `.vscode/mcp.json` |
| 3 | Tool permissions | `tapps-mcp doctor` checks `.claude/settings.json` | Claude Code only; Cursor/VS Code manage differently |
| 4 | Bootstrap done | `AGENTS.md` and `TECH_STACK.md` exist in project root | Run `tapps-mcp init` if missing |
| 5 | Python available | `tapps-mcp doctor` reports installed quality tools | Only needed for scoring/gate tools |
| 6 | CLI works | `tapps-mcp doctor` runs without error | Fallback when MCP tools unavailable |
| 7 | All checks pass | `tapps-mcp doctor` shows all green | Use `--quick` to skip slow tool version checks |

## Mapping to `tapps-mcp doctor` checks

The doctor command covers most requirements automatically:

| Requirement | Doctor check(s) | Coverage |
|-------------|-----------------|----------|
| 1. Server available | Not directly checkable (host responsibility) | Partial -- doctor checks config files exist |
| 2. MCP config | `check_cursor_config`, `check_vscode_config`, `check_claude_code_project` | Full |
| 3. Permissions | `check_claude_settings` (mcp__tapps-mcp entries) | Full (Claude Code) |
| 4. Bootstrap | `check_agents_md`, `check_cursor_rules`, hook checks | Partial -- checks artifacts, not init-run |
| 5. Python tools | `check_quality_tools` (ruff, mypy, bandit, radon, vulture, pip-audit) | Full |
| 6. CLI fallback | Doctor itself is the CLI -- if it runs, CLI works | Implicit |
| 7. Verification | Doctor output covers requirements 2-5 directly | Full |

Run `tapps-mcp doctor` as a single command to verify requirements 2-6. For requirement 1 (server visibility), check within your MCP host session.
